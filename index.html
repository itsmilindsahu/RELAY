The mathematics

Why the "best" route usually isn't the shortest one.

Relay doesn't run Dijkstra and call it a day. Because it's optimizing an average instead of a sum, the shortest-path assumption most routing algorithms lean on quietly breaks — which is the actual reason the simulator enumerates whole candidate routes instead of relaxing edges one at a time. Here's the full derivation, worked example included.

§1 The graph

Each load scatters n = 9 nodes at random positions and assigns every node a reliability r(v) ∈ [0.30, 0.99] — a stand-in for battery life, signal quality, whatever makes one phone a sturdier relay than another.

Two nodes link up if they fall inside a fixed radio range ρ, exactly the way phones discover each other over Bluetooth or Wi-Fi Direct. That's a random geometric graph: proximity decides the edges, not any central plan.

Adjacency rule G = (V, E)  ·  |V| = 9  ·  (u, v) ∈ E  ⇔  dist(u, v) < ρ

Random placement occasionally strands a cluster out of range of everything else. Relay checks this with a union–find structure: every node starts in its own set, each edge merges two sets, and if more than one set remains once all in-range edges are added, the simulator bridges the closest pair of nodes across the two nearest sets — repeating until union–find reports a single component. It's one round of the same idea behind Borůvka's algorithm, just used for connectivity instead of a minimum spanning tree.

§2 The path space

Before anything can be scored, Relay needs every reasonable way to get from source s to destination t. It enumerates simple paths — no node repeats — with a depth-limited depth-first search.

Candidate routes Paths(s, t) = { (s=v₀, v₁, …, v_k=t) : vᵢ all distinct, (vᵢ, vᵢ₊₁) ∈ E, k ≤ 6 }

That depth cap isn't cosmetic. Counting simple paths in a general graph blows up combinatorially — in the worst case, a fully-meshed set of 9 nodes gives the DFS up to 8 × 7 × 6 × 5 × 4 × 3 = 20,160 candidate routes to consider from a single source. Real mesh graphs are far sparser than that (geometry limits who's even reachable), so in practice the search tree is tiny — but the cap is what keeps the worst case bounded and the browser responsive.

§3 The objective

Every candidate path gets one number: the mean reliability of the nodes it passes through, source and destination included.

Score & objective S(P) = Σ r(vᵢ)  for i = 0…k
R(P) = S(P) / (k + 1)
P* = argmaxP ∈ Paths(s,t) R(P)  (ties → fewer hops)

Averaging instead of summing is the whole point: a raw sum rewards length for its own sake, since almost any extra node with r > 0 nudges the total up. Dividing by hop count turns the score into quality per hop — a fair comparison between a 2-hop route and a 5-hop one.

s → a → b → c → t 0.81, 0.88, 0.95, 0.91, 0.93 4 4.48 0.896 chosen
§4 Why it can't shortcut

Dijkstra and Bellman–Ford work because of optimal substructure: the best path to t must pass through the best path to some earlier node w, so you only ever need to remember the single best score seen at each node. Summed weights guarantee that — a better partial sum, extended by the same suffix, always stays better.

A mean doesn't guarantee that. Extending a path dilutes its average, and a longer partial path dilutes less per extra node than a shorter one does. So a partial route with a worse running average can end up beating one with a better running average, once both are extended by the same tail.

Counterexample partial α = (s, z): avg 0.950  ·  partial β = (s, p, q, r, z): avg 0.840
common suffix: z → w(0.10) → t(0.90)
final α = 0.725  ·  final β = 0.743 ← wins

α looked strictly better at z. After the identical bad suffix, β wins anyway, because those extra nodes made β's average harder to move. Keeping only "the best partial score per node" — the trick that makes Dijkstra fast — would have thrown β away before it ever got the chance.

Aside — why not Dijkstra with a trick?

Maximizing a mean-weight path isn't hopeless in general: it's the same family as the minimum mean cycle problem, and Karp's classic fix still applies — guess a threshold λ, subtract it from every node's reliability, and check whether any path's shifted sum is still positive using an ordinary shortest/longest-path pass; binary-search λ until it converges on the true optimum. It's real, and it's polynomial. Relay skips it because at 9 nodes and a 6-hop cap, brute-force enumeration is already cheaper than implementing a parametric search.

§5 Delivery dynamics

Picking the path is only half the simulation. Once P* is chosen, the packet animates hop by hop, and each hop's speed depends on the node it's arriving at — weaker devices visibly slow the message down instead of just silently scoring worse.

Per-hop delay delay(v) = d₀ + k · (1 − r(v))

d₀ is a fixed base latency and k scales how much unreliability costs. A node at r = 0.95 barely slows the packet; a node at r = 0.30 makes the hop visibly labor — the same number that won or lost the route in §3 is what the eye actually sees during send.
